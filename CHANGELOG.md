# Changelog

All notable changes to SpeedKalandra are tracked here. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Each release section is short and user-facing; engineering rationale lives next to the code (or in `ARCHITECTURE.md` when it shapes the design). Pre-release `v17.x` tags are listed at the end of this file as historical context only; they are no longer kept in source comments.

## [Unreleased]

### Added

- **Operational caps on the import surface (gold-standard hardening).** `RunImportService` now refuses files above `MAX_FILE_BYTES = 10 MB` before any `FileRead`, using `FileGetSize` so a 100 MB hand-crafted JSON never consumes 100 MB+ of RAM just to fail later on schema parse. `RunExportFormat` gains six caps enforced inside `ValidateSchema`, with reasoning documented next to each constant: `MAX_RUNS_PER_FILE=5000` (a multi-league archive fits; a million-entry array does not), `MAX_STRING_LEN=500` (zone names cap at ~40 chars in the live catalog; profile/patch are user-defined; 500 leaves headroom for human input without admitting 10 MB of single-field payload), `MAX_DETAILS_PER_RUN=1000` (a 4-hour run at one zone change per 15 s = 960 details, plus per-loading rows), `MAX_TOTALS_PER_RUN=200` and `MAX_ZONE_PBS=200` (77 zones today; 200 absorbs future expansion), `MAX_ACT_CHECKPOINTS=20` (6 acts today; 20 absorbs future expansion). The runId path tightens too: `_ValidateRun` now runs `RunId.IsValid` on the run's `runId` field and `_ValidatePbs` runs it on `runPbRunId`, both producing a clear "has invalid format ('value'); expected YYYYMMDD_HHMMSS with optional alphanumeric suffix" error. Empty `runPbRunId` is explicitly accepted as the well-formed "no PB yet" state. The length cap runs alongside the existing INI-breaking-char check on every textual field at the run, totals, details, and PB layers — a 50 MB profile string would otherwise pass `_FindIniBreakingChar` (no `\r\n[]`) and only fail much later on disk write. 15 new tests in `run_export_format_tests.ahk` exercise each cap (oversize array, oversize string per field, oversize map count, invalid runId format on both run and PB, empty PB runId positive control, legacy `_imported` suffix positive control). 4 new tests in a new `run_import_service_tests.ahk` cover the size gate and pre-read guards (empty path, missing file).

- **Defensive smoke tests for the UI under realistic worst cases.** `CompactLayoutWidget` gains two tests: `estimate_text_w_handles_200_char_zone_name_linearly` proves the `chars × fontSize × 0.6` estimator returns a sane, positive value at 200 characters (the realistic worst-case zone name from a hand-edited INI or a future game expansion with verbose location strings), and `on_zone_entered_accepts_200_char_zone_name_without_throwing` proves the bus handler stores the value verbatim — the rendering side (font shrink, clipping) is delegated to the lazy `_Refresh` tick which is guarded by `if !this._gui return` and can't be exercised headless. `RunStatsPlotBuilder` gains `build_aggregates_100_plus_zone_entries_without_overflow` (120 zones × 60 s = 7,200,000 ms aggregate) and `build_aggregates_100_plus_loading_events_without_overflow` (150 transitions × 5 s = 750,000 ms aggregate) — proves the in-memory snapshot path (which has no `MAX_DETAILS_PER_RUN` cap because it's not gated by the import schema) sums correctly without overflow and produces exactly one detail row per input entry.

### Changed

- **`GAME_EXECUTABLES` consolidated into `GameProcesses.POE2_EXECUTABLES`.** `FocusAutoPauseService` and `LoadingDetectionService` previously each declared the same hardcoded list of PoE2 executable names with a "keep in sync" warning in both files. The list now lives in `src_v2/domain/game_processes.ahk` and both services consume it directly. 6 new tests in `game_processes_tests.ahk` lock the contract (non-empty, every entry `.exe`, no duplicates, canonical standalone and Steam builds present).

- **`build-dist.ps1` release gate now genuinely waits for the AHK suite.** The previous form `& $ahkExe "tests_v2\run_tests.ahk"` was a silently broken gate when invoked from PowerShell 5.1 (which is what the `build-dist.bat` wrapper uses by default). AutoHotkey64.exe is Windows-subsystem, not console-subsystem, and PS 5.1 does not wait implicitly on GUI EXEs called via `&` — it is fire-and-forget. Empirical evidence: the build returned and produced the zip while the AHK process was still running the 50-second suite; the test summary printed *after* the zip was on disk. A null-defensive `$exitCode = $null ? 0 : raw` line, originally added because the CI runner sometimes returned `$null` from native processes, made the bug invisible by mapping the no-wait `$LASTEXITCODE = $null` to a treated-as-success 0. The gate is now `Start-Process -FilePath $ahkExe -Wait -PassThru -WindowStyle Hidden` plus `$testProc.ExitCode` (guaranteed populated by .NET's `Process.ExitCode` after `WaitForExit`, no null-defensive fallback needed); `-NoNewWindow` is deliberately omitted because it caused the original 19-minute hang on the GitHub Actions runner (inherited stdio handles leaving `WaitForExit` blocked on a pipe that never closes). A second layer of defense compares `tests_v2/tests_output.log`'s LastWriteTime against `$gateStartTime` captured before AHK launched — if AHK reports exit 0 but the log is stale, the build aborts loud ("the runner did not produce a fresh log during this build"). The CI workflow (`shell: pwsh`, PowerShell 7) keeps the `&` form because PS 7 *does* wait on GUI-subsystem EXEs invoked via `&`; the local-only nature of the bug is documented in the new comment block above the gate.

- **Test-suite count refreshed in `tests_v2/README.md` and `src_v2/README.md` ("over 1800 tests" instead of "over 1500").** The coverage table also gains `RunImportService (size gate)` under stateful services.

### Documentation

- **`ARCHITECTURE.md` §6 documents the terminal lifecycle and the Dispose-vs-Stop policy explicitly.** Adds a paragraph stating that `Stop()` is terminal (`Start` after `Stop` throws, `Stop` itself is idempotent, reload is a new process via the tray menu's Reload item, the `OnExit` handler in `speedkalandra.ahk` sends defensive modifier key-ups before invoking `Stop`), and a paragraph documenting the audit conclusion that `Stop()` already releases every service with an external resource (AHK SetTimers, AHK `Hotkey()` registrations, file-tail state, the bus interceptor, the three widget Gui handles) — services that only subscribe to the bus expose `Dispose()` for symmetry but are not invoked from `Stop()`, because the bus itself is dropped when the process exits or a new instance is constructed. Calling those Dispose()s would be ceremony with no observable effect.

- **`ARCHITECTURE.md` §5 lists the import caps inline on the `RunImportService` row.** A reviewer reading the services table now sees the same numbers that live in `RunExportFormat` (`MAX_RUNS_PER_FILE=5000`, `MAX_STRING_LEN=500`, `MAX_DETAILS_PER_RUN=1000`, `MAX_TOTALS_PER_RUN=200`, `MAX_ZONE_PBS=200`, `MAX_ACT_CHECKPOINTS=20`) plus the file-level cap (`MAX_FILE_BYTES=10 MB`) and the explicit mention that `RunId.IsValid` is enforced on both `runId` and `runPbRunId`.

- **`ARCHITECTURE.md` §8 corrects the Undo/PB behaviour.** The previous text claimed "Personal bests are not reverted" after Undo, but the implementation has rebuilt PBs from the remaining history via `PersonalBestService.RebuildFromHistory` since Fase 7 of the prior roadmap. The text now describes the actual behaviour (file deleted + PBs rebuilt + tray item auto-expires after 60 s).

- **`SECURITY.md` Anti-cheat / TOS bullet replaces the absolute "does not inject or send inputs" with the actual behaviour.** The app does emit defensive `{Ctrl up}{Alt up}{Shift up}{LWin up}{RWin up}` at process exit and `{Ctrl up}{Alt up}{Shift up}` before opening dialogs from a hotkey, to avoid the AHK "stuck modifier" failure mode. Those key-ups are not gameplay commands. The disclaimer now mentions them explicitly.

- **`src_v2/README.md` fixes the `run_state_repository.ahk` path comment.** The map showed `data/run_state.ini + zone_totals.txt` but the actual paths have been `speedkalandra.ini [RunState] + speedkalandra_zones.txt` since the split-file persistence change.

- **`app.ahk` event-tracing comment corrected.** The previous comment claimed the interceptor was registered "BEFORE any service subscribes" but `EventTraceLogger.__New` only constructs the object; `AddInterceptor` runs in `tracer.Start()` called from `SpeedKalandraApp.Start()` and only when `cfg.eventTracingEnabled`. The comment now states that explicitly and notes that the `RunStarted{hydrated:true}` event emitted during `runService.Hydrate` at the end of `__New` is NOT captured by the tracer.

- **`CONTRIBUTING.md` aligns its "Read-only" and "What I won't merge" bullets with the same key-up nuance.** The merge policy now says "sends gameplay commands" instead of "sends inputs" so the existing defensive key-up doesn't read as a self-violation.

### Added

- **`.gitattributes` fixing line endings and binary handling.** `*.ahk`/`*.ps1`/`*.bat`/`*.cmd` pinned to CRLF (Windows-native tooling — the AHK error dialog and Notepad++ expect it; the build/tray flows only run on Windows); `*.md`/`*.csv`/`*.yml`/`*.yaml`/`*.json` pinned to LF (the cross-platform default, also what GitHub renders cleanest); `*.ini` and images/archives marked binary so git never tries to text-normalize a UTF-16 LE BOM file that `IniWrite` produced (no INI is committed today, but the rule is a guardrail for any future template/fixture). Existing files were intentionally **not** mass-renormalized (`git add --renormalize .` was not run) — only new commits respect the new rules. A separate PR can renormalize the whole repo at once if desired; the diff would be massive and worth handling on its own, not bundled here.

### Fixed

- **Hotkey-interrupted zone visit polluted PB totals (artificial sub-second zone PBs).** Pressing `Ctrl+Alt+F` to finalize a run shortly after entering a new zone left that zone with an artificially low PB candidate equal to the few seconds the player spent there before the hotkey. The same zone, in a normal run that crossed the transition out of it, would land 60–120 s — the interrupted-visit number replaced it because `RunSnapshotSaver` was passing the factual `zoneTotals` straight to `PersonalBestService.UpdateFromRun`, and the factual total at that point included the just-flushed elapsed of the interrupted visit. The fix tracks per-visit elapsed (`ZoneTrackingService._currentVisitMs`, accumulating in `_FlushActive` and resetting on every transition / lifecycle end), reads it from `RunSnapshotSaver.Save` via the new `GetCurrentVisitMs()` query in the `OnBeforeFinalize` hook (after `TimerStopped` flushes the visit but before `RunCompleted` publishes), subtracts it from a copy of the zone totals (`pbZoneTotals`), and passes that copy to `UpdateFromRun`. The factual `zoneTotals` is unchanged — the run's history file and post-run plot still show the full visit time as a historical fact; only the PB candidates exclude it. Permissive case preserved: if the zone had a prior complete visit in the same run (closed via transition) plus the interrupted one, only the interrupted visit's time is subtracted, the prior closed visit still counts as a PB candidate. Single-visit interrupted zones drop out of PB-eligible totals entirely (negative remainder → zone removed instead of recorded with a negative or zero PB). The `(interruptedZoneName, interruptedZoneVisitMs)` pair is persisted into the run's `[meta]` section by `RunHistoryRepository`, so `PersonalBestService.RebuildFromHistory` (the path that runs after `UndoLastSave` and after `Delete` from `RunHistoryDialog`) mirrors the same discount — Undo lands on the exact same PB as the original save, never on the inflated pre-fix one. Legacy runs persisted before this field existed load with the defaults `""` / `0` and skip the discount cleanly (no regression on previously-saved history). The visit accumulator is in-memory only by design (see `KNOWN_ISSUES.md` for the narrow crash + same-zone + hotkey edge case). 37 new tests: 14 on `ZoneTrackingService` (per-visit accumulation, pause/resume preservation, reset on every lifecycle path, finalize-sequence end-to-end), 8 on `RunSnapshotSaver` (discount applied, zone dropped when visit equals or exceeds total, keys injected into buildResult, cancelled runs don't call `UpdateFromRun`), 6 on `RunHistoryRepository` (INI round-trip, legacy default, defensive rejection of `\n` / `[` in `interruptedZoneName`), 7 on `PersonalBestService.RebuildFromHistory` (mirrored discount, drop-on-equal, defensive on exceed, no-op on empty/zero keys, only matching zone discounted, back-compat with legacy runs), and 2 integration tests in `speedkalandra_app_integration_tests.ahk` (original bug reproduction + permissive case) wiring the three layers through the real composition root.

- **Zone internal IDs leaking as human names (engine `G1_2` shown/saved instead of `Clearfell`).** PoE2's `Client.txt` emits `[SCENE] Set Source [<raw>]` with `<raw>` being either a human zone name ("Mud Burrow") or the engine's internal id ("G1_3"). The old `LogMonitorService` copied the raw text straight into `Events.ZoneChanged{zoneName: raw, sceneId: raw}`, so when the engine emitted the id, every downstream consumer keyed its data by the id: `ZoneTrackingService._totals["G1_3"]`, `RunStatsPlotBuilder._AddZoneDetails` finding `FindByName("G1_3")` empty (fell back to category="mapa", act=0, label="G1_3"), `RunHistoryRepository.Save` writing `[totals] G1_3=ms` to `data/runs/{runId}.ini`, and `PersonalBestService._zonePbs["G1_3"]` in `personal_bests.ini`. `GetActTotals()` silently dropped the zone from per-act tallies because `FindByName` of the id failed. The fix resolves the raw text to the canonical human name from `ZonesCatalog` at the single publisher (`LogMonitorService._ResolveZoneToHumanName`): try `FindByName` first (catches human-name lines, recovers canonical case), then `FindById` (catches engine-id lines, returns the catalog's display name), then pass through the raw text for unknown zones (a new game patch or randomized instance still surfaces in the overlay/history — just without act/isTown metadata). `Events.ZoneChanged.sceneId` continues to carry the raw text for diagnostics. The catalog is an optional constructor parameter on `LogMonitorService`; tests/headless setups without one preserve the pre-fix behaviour. The legacy `"You have entered <zone>"` parse branch is also routed through the resolver — not observed in current PoE2 `Client.txt` (every real-world `ZoneChanged` fires from `[SCENE]` today), retained defensively in case the engine reintroduces the line. 8 new tests across `log_monitor_service_tests.ahk` (resolution by id, resolution by name with case recovery, fallback for unknown zones, no-catalog passthrough, constructor type-check) plus 1 end-to-end guardrail in `speedkalandra_app_integration_tests.ahk` that proves `[SCENE] Set Source [G1_3]` lands in `zoneTracker._totals["Mud Burrow"]` (not `["G1_3"]`). Historical run files on disk that were saved with raw-id keys keep working (Load is unchanged); the user can `Reset PBs` or import-rebuild to regenerate clean PB keys from new runs.

- **INI-breaking characters in textual fields now rejected at import and on save.** Run files (`data/runs/{runId}.ini`) and the personal-bests INI rely on `\r`, `\n`, `[` and `]` having reserved structural meaning. Before this fix nothing stopped a hand-edited or maliciously crafted import payload from putting any of those characters into `profile`, `patch`, `firstTs`, `runId`, a zone-name key under `totals`, a detail row's `category`/`label`/`note`/`timestamp`, the PB `runPbRunId`, or a `zonePbs` zone key. A single `\n` would silently split a row across two INI lines on the next save, making the `[details].count` value mismatch the actual rows and corrupting `Load`. The fix is two layers: (1) `RunExportFormat.ValidateSchema` now scans every textual field on every run plus the PB block, and reports `"runs[N].fieldName contains INI-breaking character (\\n)"` so the user sees which field needs cleaning before the import touches disk; (2) `RunHistoryRepository._SerializeBuildResultToIni` defensively throws `ValueError` on the same character set, routed through the existing `WarningSink` so `Save` returns `false` and emits a warning identifying the failed `runId` — protection against a future parser regression smuggling bad text in through some other code path. Safe punctuation (`=`, `;`, `|`, parens, long strings, Unicode) is explicitly preserved by a round-trip test. 17 new tests: 9 covering each rejected field/character combination at the import layer (`run_export_format_tests.ahk`), 5 round-trip tests confirming safe characters survive Save/Load unchanged, and 3 defensive-throw tests confirming a bad field surfaces through the `WarningSink` instead of corrupting the file (`run_history_repository_tests.ahk`).

- **Removed accidental Exit hotkey (`Ctrl+Alt+Q`).** The entry script registered `^!q::ExitApp()` as a global hotkey. On ABNT2 and other international keyboards `AltGr+Q` is reported by Windows as `LCtrl+RAlt+Q`, which AHK's `^!` matches exactly — pressing AltGr+Q during a run closed the app and lost in-progress state. The tray menu "Exit" item already covers the use case, so the global hotkey was redundant and unsafe. Removed entirely; tray-menu Exit is the documented way to close the app.

- **`RunHistoryRepository.Save` is now atomic.** The previous implementation made roughly N direct `IniWrite` calls against `data/runs/{runId}.ini` (meta + totals + checkpoints + details, all sequential, each one a separate file open/modify/close round trip). A crash anywhere in that sequence — PC freeze, BSOD, mid-finalize close, antivirus quarantine grabbing the file — left a half-written run on disk: `[meta]` complete, `[totals]` missing, `[details]` count out of sync with the actual rows. The new implementation builds the full INI in memory and writes it through `AtomicWriter` (`.tmp` + `FileMove`), so the destination only ever sees a fully-formed file or no change at all. Same byte-compatible UTF-16 LE BOM output that `IniWrite` used to produce, so existing runs on disk and the `Load` path stay unchanged. The pure `_SerializeBuildResultToIni(buildResult, currentRunId, totalMs)` helper that builds the content is testable in isolation, and Save now also routes its failure path through the injected `WarningSink` instead of returning `true` regardless of what happened on disk. Two new tests cover the atomicity contract: no `.tmp` orphan after a successful save, and `Save` returns `false` + warns on writer failure.

- **Hydration ordering bug (RunStatsRecorder silently dropped hydrated runs).** `SpeedKalandraApp.__New` used to call `RunService.Hydrate` mid-construction. When the loaded state had an active run, `Hydrate` published `Evt.RunStarted{hydrated:true}` before `RunStatsRecorder` (and other downstream subscribers) had been created. Finalizing the hydrated run later produced a snapshot with `runId=""`, which `RunHistoryRepository.Save` silently rejected — the run was lost. `Hydrate` is now deferred to the end of `__New` after `_WireEventHandlers()`, and `ZoneTrackingService._OnRunStarted` respects the `hydrated:true` flag so the in-flight event no longer wipes the totals that were just restored from disk. Regression tests added in `zone_tracking_service_tests.ahk` and `speedkalandra_app_integration_tests.ahk`.
- **`LoadingTotalsService` wiped hydrated `_totalMs` on `RunStarted{hydrated:true}`.** Sibling of the hydration-ordering bug above: the composition root hydrates the loading total from disk via `loadingTotals.Hydrate(loadingMs)`, but `_handlerRunStarted` was hard-wired to `Reset()` and so the deferred `RunService.Hydrate()` event zeroed the value immediately after restoration. Loading time of an in-progress run was lost on every reload. Handler now goes through `_OnRunStarted(data)` which honours the `hydrated:true` flag, mirroring `ZoneTrackingService`. Regression tests `run_started_with_hydrated_flag_preserves_total_ms` and `run_started_without_hydrated_flag_wipes_total_ms` added in `loading_totals_service_tests.ahk`.
- **`LoadingDetectionService` false positives from non-game windows.** The default window provider matched the substring `"Path of Exile 2"`, which also matched Chrome tabs on the PoE2 wiki and Discord channels with that text. The HUD scanner then sampled pixels from the wrong window. Replaced with an `ahk_exe` scan over the same canonical list `FocusAutoPauseService` uses (`PathOfExile2Steam.exe`, `PathOfExile2_x64.exe`, etc.), and lock every follow-up `WinGetMinMax` / `WinGetPos` to the resolved HWND via `ahk_id`.
- **`UndoLastSave` left personal bests pointing at the deleted run.** Inconsistent with `RunHistoryDialog.Delete`, which already rebuilt PBs. `UndoLastSave` now invokes a new `_RebuildPbsFromHistory` helper after a successful delete (mirrors the dialog), so deleting the most-recent run via the tray immediately purges any PBs the deleted run contributed to.

### Changed

- **Simplicity audit + residual cleanup (Fase 11).** Qualitative review of the codebase against GSG §10's overengineering signals (formal class names, pass-through abstractions, abstractions without tests proving value, comments a better name would solve, patterns applied because "professional", `app.ahk` growing without organic extraction). The composition root, the four extracted collaborators (`BootPrompts`, `RunSnapshotSaver`, `RunStatePersister`, `LiveReconfigurationHandlers`) and every service were inspected; the conclusion is that the current shape pays for itself — each abstraction has a documented purpose and a test that exercises behaviour rather than constructor-only smoke. What this commit changes is the residue the Fase 12 strip missed: six files still carried historical phase/version/bug tags (`v17.13`, `v17.14`, `v17.15.3`, `v0.1.2 (audit #19)`, `v0.1.0 Phase 5`, `Phase 9.7`, `Phase 9.10`, `Item 1`, `Item 2`, `Smoke fix Turn 2`) in source comments — those are removed without changing behaviour, per GSG §11 (history belongs in the changelog, not the source). While in `steve_layout_widget.ahk` the audit also surfaced genuine dead code left over from the bar removal: the `COLOR_MAPA` / `COLOR_LOADING` / `COLOR_CIDADE`, `WHALE_IMG_PATH` / `WHALE_X` / `WHALE_Y` / `WHALE_W` / `WHALE_H` / `WHALE_GAP` static constants, the `_whaleLoaded` instance field, and the `BAR_TIMER_GAP` constant were declared but never read — the bar and the decorative whale image were both removed in earlier passes and left these declarations stranded. The `loadingTotals` constructor parameter (and the matching `_loadingTotals` field) on `SteveLayoutWidget` was stored at construction but never read either, so it was removed from both the signature and the call site in `app.ahk`; no tests exercised `SteveLayoutWidget` so no test changes were needed. Net diff: ~70 source-comment lines and ~15 dead declarations gone, zero behaviour change, suite stays at 1808/1808. The audit notes that `LiveReconfigurationHandlers` has formal-looking cohesion (three handlers grouped because they share `cfg`/`log`/`headless` constructor args rather than because they share a domain) but inlining it would require rewriting 16 unit tests against `SpeedKalandraApp` integration — cost beats benefit, the class stays. `EventBus` was reviewed and stays: the pre-publish-hook fix in Fase 2, the `_AssertWired` boot-time check, and the `EventTraceLogger` diagnostic all exist specifically because there is a bus, and removing it would re-introduce hard coupling at the composition root.

- **Tag-triggered release workflow + CI validation of the build script + hang-defence for the AHK suite (Fase 8).** Three pieces close the release pipeline: `.github/workflows/release.yml` (new) runs on pushed `v*` tags (and via `workflow_dispatch` against an existing tag), installs AutoHotkey v2 on a Windows runner, runs `tests_v2/build/test_build_dist.ps1` as a smoke check on the build script itself, then runs `build-dist.ps1 -Compile -Zip -Force` (which already runs the full AHK suite as a release gate — a red suite aborts packaging) and creates a **draft** release on GitHub with `SpeedKalandra-${tag}.zip` and `SpeedKalandra-${tag}.zip.sha256.txt` attached. The maintainer reviews the draft and publishes manually; auto-publish was rejected because a wrong tag would require force-pushing to recover. The workflow uses the `gh` CLI (preinstalled on GitHub-hosted runners) instead of a third-party action so the release pipeline does not add a supply-chain dependency. Both CI workflows pin AutoHotkey to version 2.0.19 explicitly (`choco install autohotkey --version=2.0.19`) instead of taking the latest from chocolatey — the latest at the time of writing (2.0.24) reproducibly hangs one integration test (`complete_run_flow_from_start_to_finalize_to_undo`) under non-interactive CI sessions with no error reaching `OnError`, stderr, or the log. The test passes locally on 2.0.19 and the rest of the suite passes on 2.0.24, so the failure is narrow and the pin keeps the CI signal honest until that one test can be isolated into a smaller reproducer; bumping the pin to 2.0.20+ is a tracked follow-up. `.github/workflows/test.yml` gains a `Validate build-dist script` step running `test_build_dist.ps1` on every push and PR — catches leakage regressions (a personal-data sentinel slipping past the filter into the zip), SHA256 sidecar format drift (the verification command in RELEASE.md would silently break for end users), and `-AhkPath` resolution regressions. Both workflows also gain explicit `timeout-minutes` caps on every long-running step (8 minutes on the AHK test suite, 3 minutes on the build-dist validator, 15 minutes on the release build, 25 minutes job-level cap): without these, a hung test could silently consume up to GitHub's 6-hour default, and the `Upload test log` step (`if: always()`) would never run because the prior step never ended — with the timeouts, a hang fails the step loud at a sane bound and the log artifact is still preserved for diagnosis. The test step also swaps `Start-Process -Wait` for a direct `&` call (the previous form can sit waiting on inherited handles even after the child process has exited — a known Windows-runner quirk that produced a 19-minute zero-output hang on the first CI build of these tests) and passes `/ErrorStdOut` so AHK parse errors hit stderr instead of a modal dialog that would itself hang the runner. `tests_v2/run_tests.ahk` installs an `OnError` handler **before any `#Include`** that captures runtime errors during global-scope code (a static initializer, a top-level setup line) which would otherwise pop a MsgBox and hang non-interactive sessions — the handler writes the error to a new `tests_v2/tests_boot.log` (separate from `tests_output.log` because `TestReporter.Init()` deletes the latter on entry) plus stderr, then `ExitApp(2)`. The boot log also receives a header on every run (AHK version + working dir + env state) and a `REG: ClassName` line for every successful `TestRegistry.Register` plus a `RUN:` / `OK:` pair around every test execution in `TestRunner._RunOne`, so a future hang can be diagnosed from the artifact alone — the last `RUN:` without a matching `OK:` names the test that killed the process. Both logs are uploaded as workflow artifacts via `if: always()`. `build-dist.ps1` gains a `-AhkPath` parameter with a new `Resolve-AhkPath` helper covering: explicit path (must exist; failure does NOT fall through to the discovery cascade, so a misconfigured CI runner fails loud instead of silently falling back to an arbitrary system AHK), standard install locations (`Program Files\AutoHotkey\v2\...`, x86 variants, `LOCALAPPDATA\Programs\...`), and `PATH` via `Get-Command`. `test_build_dist.ps1` gains a scenario B verifying that an explicit `-AhkPath` pointing at a non-existent file aborts the build before any side effect (no dest dir is created, exit code is non-zero); without this guard, a future "convenience" refactor that fell back to standard paths on explicit miss would silently re-introduce the CI-misconfiguration trap. New `RELEASE.md` documents the full release path: how to create a release (tag + push), the manual `workflow_dispatch` route, local-only build with `-SkipTests`, the three release gates (AHK suite green, `test_build_dist.ps1` green, AHK v2 installed), and a troubleshooting section covering the common failure modes.
