# Changelog

All notable changes to SpeedKalandra are tracked here. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Each release section is short and user-facing; engineering rationale lives next to the code (or in `ARCHITECTURE.md` when it shapes the design). Pre-release `v17.x` tags are listed at the end of this file as historical context only; they are no longer kept in source comments.

## [Unreleased]

### Changed

- **`MAX ACT FILTER` no plot de Run Statistics passa a cortar dados acima do ato, não filtrar runs inteiras.** O dropdown vinha rotulado como "MAX ACT FILTER" mas implementava semântica de **min act**: `Act 3+` significava "incluir apenas runs cujo `maxActReached >= 3`" (filtra runs inteiras pelo que cada uma alcançou), e o filtro só afetava o line chart — KPIs do header, total no header e barra "Current run distribution" mostravam a run completa. Resultado: ao escolher `Act 1+` numa run que chegou ao meio do Ato 2, o tempo gasto no Ato 2 parcial continuava aparecendo em todas as superfícies da view. A semântica nova é "cortar dados acima do ato N": ao escolher `Act 2`, o plot inteiro (KPIs, current-run bar, line chart) reflete apenas dados de Acts 1 e 2; tempo gasto em Acts 3+ é descartado da view (não dos dados em disco, que continuam intactos). Runs históricas com `maxActReached < N` continuam aparecendo no chart — mostram só o que têm, em vez de serem excluídas.

  Mecanismo: novo método estático `RunStatsPlotBuilder.FilterByMaxAct(data, maxAct)` retorna uma cópia do `data` com `details` filtrado por ato (extraído do `note` via regex `(?:Ato|Act)\s+(\d+)`, aceitando ambos os formatos legacy/atual), e `totals`/`totalMs` recalculados sobre os details retidos. Três políticas de over-include preservam dados que o filtro não consegue ancorar com confiança: (a) details com `category=morte` sempre passam (mortes não carregam ato no snapshot atual — ver BACKLOG B2 pra trilha da feature que adicionaria), (b) details cujo `note` não parseia como ato sempre passam (zonas fora do catálogo, runs legacy), (c) `maxActReached` no result Map preserva o valor original — descreve a run, não a view filtrada. Helpers `_DetailPassesMaxAct` (predicate) e `_ShallowCloneData` (clone para evitar mutação do input — o dialog cacheia o build unfiltered e reaplica o filtro a cada mudança de dropdown).

  Loadings agora carregam `note: "Act N"` derivado da `toZone` via catálogo, dentro de `_AddLoadingDetails`. A atribuição pela zona de destino é semanticamente correta — um loading **para** o Ato 2 pertence ao orçamento do Ato 2, não do Ato 1. Loadings cuja `toZone` não está no catálogo ficam com `note=""` e caem no over-include. Efeito colateral: o granularity "By act" do line chart agora inclui loadings no total de cada ato (mais preciso — o tempo "do Ato 1" deveria incluir loadings durante o Ato 1; antes ficavam de fora).

  Lista do dropdown: `["All", "Act 1", "Act 2", "Act 3", "Act 4", "Interlude"]` (era `["All", "Act 1+", ..., "Act 10+"]`). Acts 5-10 nunca existiram em produção — o catálogo (`data/zones.csv`) tem só Acts 1-4. "Interlude" é placeholder: hoje mapeia pro sentinel `maxAct=999` que funciona como no-op (igual a "All"), porque o tracker não distingue cruel/interlude do normal — esse trabalho está no `BACKLOG.md` item B1 (cruel/interlude tracking pipeline). O sentinel é validado por teste pra que uma futura refactor não rompa o contrato silenciosamente. Sinal `+` removido do label conforme pedido.

  Aplicação consistente: `RunStatsPlotDialog._ShowWithData` aplica `FilterByMaxAct(data, this._maxActFilter)` ao current data ANTES de `_BuildGui`, então KPIs/header total/current-run bar refletem o filtro. `_CollectRunsForChart` aplica `FilterByMaxAct` a cada run histórica carregada (já não há mais filtro "drop run if maxActReached < N"), e a política de carga muda: quando o filtro está ativo (`maxAct > 0`), sempre carrega o INI completo da run (não o summary), porque o summary só tem `totals` pré-agregados e não permite recortar por ato post-hoc. Quando o filtro é "All", o summary continua sendo o caminho rápido sob granularity "run" — preserva a performance da rota mais comum.

  Renames: `MIN_ACT_LABELS` → `MAX_ACT_LABELS`, `_minActFilter` → `_maxActFilter`, `_OnMinActFilterChanged` → `_OnMaxActFilterChanged`, ctrl key `"dropdown_min_act"` → `"dropdown_max_act"`. Dois helpers novos `_MaxActFilterValueFromIndex` / `_MaxActFilterIndexFromValue` mantêm o mapeamento idx↔value simétrico entre o handler e o `Choose<N>` inicial (`idx 1 → 0 All`, `idx 2..5 → 1..4 Acts`, `idx 6 → 999 Interlude`).

  ToS compliance (GSG §18): mudança puramente de UI/agregação. Zero leitura de game files novos, zero simulação de input, zero rede. **No risk** under PoE2 Terms of Use clauses 7(b–f, i). GSG §17 anti-regression review: 24 items revisados, nenhum violado; em particular, `maxActReached` continua sendo derivado e persistido do mesmo jeito (o filtro só passa a usar pra outra coisa), a round-trip de persistência não é tocada, o schema do INI não muda, e os legacy notes `"Ato N"` continuam aceitos pelo regex de extração. 17 testes novos em `run_stats_plot_builder_tests.ahk`: 3 cobrem o `note` de loadings (atribuição via `toZone` no catálogo, vazio quando a zona é desconhecida, vazio quando `toZone` está ausente), e 14 cobrem `FilterByMaxAct` (no-op em `maxAct=0` e negativo, drop de details acima do max, retenção no boundary `act == maxAct`, mortes bypassam, details sem ato parseável bypassam, loadings com act ≤ maxAct sobrevivem, loadings com act > maxAct caem, totals/totalMs recalculados, `maxActReached` e metadata preservados, input não-mutado, e o sentinel 999 do Interlude como no-op).

- **Unified layout control: 3 hotkey actions → 1 `CycleLayout`, `ToggleOverlay` removed.** The previous `ToggleMicroLock` / `ToggleSteveLock` / `ToggleOverlay` trio collapsed into a single `CycleLayout` action that walks `STEVE → COMPACT → MICRO → STEVE` in a fixed loop. The two `Toggle*Lock` actions gave the user two separate hotkeys for the same conceptual choice ("pick a layout"); a speedrunner usually picks a layout pre-run and stays in it, and the rare mid-session swap is now one repeated keypress instead of two competing bindings. `ToggleOverlay` (hide/show overlay) was removed entirely — the hover-dim feature in `OverlayInteractionService` already drops opacity to ~10% when the mouse passes over the overlay, which covers the underlying "the overlay is in the way of an item drop" need without the cognitive load of a hide-state the user has to remember to toggle back.

  Default hotkey for `CycleLayout` is `^F8` (the slot `ToggleSteveLock` used to occupy by default) so a fresh install lands on a working bind. Returning users keep their muscle memory via migration in `SettingsRepository._LoadHotkeys`: the legacy `ToggleSteveLock` bind takes precedence over `ToggleMicroLock`, both override only when `CycleLayout` is absent from the INI (an explicit user bind always wins), and an empty legacy bind never shadows the default — a user who cleared both lock toggles in the old Settings dialog doesn't silently lose the new `^F8`. `ToggleOverlay` is discarded silently from any INI it appears in. On the next Save, `_SaveHotkeys` writes only `CycleLayout` (the `_SyncMapSection` helper already removes obsolete keys), so the legacy entries disappear from disk without a separate cleanup pass.

  Touches: `Commands` (three constants out, `CycleOverlayLayoutRequested` in; `SetOverlayModeRequested` retained as the programmatic API), `HotkeyService.ActionToCommand` (9 → 7 entries), `OverlayModeService` (replaced `ToggleMicroLock` / `ToggleSteveLock` methods with a single `CycleLayout`; the `_SyncWindowFlags` invariant — mutually exclusive lock flags persisted into `cfg.window` — is preserved so the Hydrate path still reconstructs the right mode on the next boot), `SpeedKalandraApp.ToggleOverlay()` (deleted along with its `_WireEventHandlers` subscribe), `AppSettings._DefaultHotkeys` (9 → 7), `SettingsRepository._LoadHotkeys` (migration + ToggleOverlay drop). The `Evt.OverlayModeChanged{mode, prevMode, locked, steveLocked}` payload shape is unchanged — downstream subscribers (notably the persistence path that mirrors locked flags into `cfg.window`) keep working without edits.

  ToS compliance (GSG §18): pure overlay/hotkey refactor. Zero game-file reads, zero input simulation, zero network. **No risk** under PoE2 Terms of Use clauses 7(b–f, i). GSG §17 anti-regression review: all 24 items reviewed, none violated; in particular, hotkey persistence stays in `[Hotkeys]` with the same Save-removes-obsolete behaviour, `OverlayModeService` still subscribes BEFORE `Hydrate` is called from the composition root, no FIFO-dependent flows touched, and the cycle handler stays a pure state machine with no GUI side effects (the TrayTip is silent on Mute, matching the previous `_OnSetModeRequested` behaviour). ~17 new/changed tests across four files: `overlay_mode_service_tests.ahk` rewritten end-to-end (10 cycle tests covering every transition plus the mutually-exclusive-flags invariant, 2 command-subscriber tests for `CycleOverlayLayoutRequested`); `hotkey_service_tests.ahk` updated for the 9→7 collapse plus a new `cycle_layout_maps_to_cycle_command` lock test; `app_settings_tests.ahk` updated (`defaults_hotkeys_includes_seven_actions` pins both the count and the explicit absence of the three removed actions); `settings_repository_tests.ahk` adds 7 migration tests — each legacy bind migrating individually, Steve precedence, explicit CycleLayout wins, ToggleOverlay discarded, empty legacy ignored, and a full round-trip proving the legacy keys don't resurrect on disk after a Save.

### Added

- **Run-outcome banner (closes the "did it save?" feedback gap).** A transient banner now appears top-centre on the overlay for ~4 s after every Finalize / Cancel / Reset of an active run, announcing what actually happened: `SAVED · 1:12:34 · PB` (green) / `SAVED · 1:12:34` (soft green) / `DNF · 1:12:34` (amber) / `TOO SHORT · 02:42 · NOT SAVED` (orange) / `RESET · NOT SAVED` (grey). The colour comes from `Theme.Color` (`goodStrong` / `good` / `warn` / `accentSoft` / `subtle`), so the palette stays consistent with the rest of the overlay and a future theme tweak picks the banner up for free. The window is click-through (`WS_EX_TRANSPARENT`), `+AlwaysOnTop`, and auto-clears either on the SetTimer fire or immediately on the next `Evt.RunStarted` so it can never sit on top of a fresh run's HUD. Opt-out via the new Settings → DISPLAY checkbox "Show run-outcome banner after each run" (default on so the feedback gap is closed for new users); the flag is live-reloadable through the new `Evt.ShowOutcomeBannerChanged` event — if the user turns it off while a banner is on screen, the widget hides it immediately. Two events orchestrate the flow: `RunSnapshotSaver` now takes an optional `bus` in its constructor and publishes `Evt.RunOutcomeReported{outcome, durationMs, runId, pbChanged}` from each of its three exit paths (saved, dnf, too_short) with a single internal helper that short-circuits on missing bus so existing test setups don't have to wire one. `RunService.ResetRun()` captures `runMs` **before** calling `timer.Reset()` and publishes the same event with `outcome:"reset"` (gated on `wasActive` — a reset on idle state has no user-meaningful outcome to report, only the lifecycle `RunReset` still fires for downstream state-clearers). `RunOutcomeBannerWidget` (`src_v2/ui/run_outcome_banner_widget.ahk`, ~290 lines) is standalone — it deliberately does NOT extend `WidgetBase` because the banner isn't user-positionable, doesn't participate in mode switching (NORMAL / COMPACT / MICRO), and doesn't need scale, Ctrl-drag or the highlight border. The widget is constructed once at boot in `app.ahk`, added to `_AssertWired`, and disposed in `Stop()` so a Reload cycle doesn't leave a stale Gui handle behind. Persistence: new `[Display].ShowOutcomeBanner` INI key, default `"1"` so an upgrade over an older INI keeps the banner on (opt-out semantic preserved); `AppSettings.showOutcomeBanner` field, defaults `true`. ToS compliance (GSG §18): the widget reads NO game files, simulates no input, makes no network call — it's a pure overlay surface driven by tracker-internal events. **No risk** under PoE2 Terms of Use clauses 7(b–f, i). GSG §17 anti-regression review: all 23 items reviewed, none violated; in particular the run-completion save path is untouched (saver still runs in `OnBeforeFinalize` / `OnBeforeCancel`, threshold and PB-eligible-totals logic unchanged), `Stop()` disposes the new widget alongside the other GUI surfaces, and the new event types follow the existing `Evt.*` naming. ~30 new tests across six files: 5 in `run_outcome_banner_widget_tests.ahk` (constructor type-checks, three subscribers wired, message formatter per outcome, colour selection, duration formatter for HH:MM:SS / MM:SS / negative-clamp, cfg.showOutcomeBanner opt-out, RunStarted clears, ShowOutcomeBannerChanged{newValue:false} hides while {newValue:true} stays inert, defensive paths for malformed payloads, Dispose idempotency); 7 in `run_snapshot_saver_tests.ahk` (outcome publishing for the three Save() exit paths, pbChanged round-trip, silent when bus is missing, silent when `runHistory.Save` returns false); 5 in `run_service_tests.ahk` (ResetRun publishes outcome when active, captures pre-reset duration, pbChanged always false, idle reset stays silent, publish order RunReset → RunOutcomeReported); 2 in `app_settings_tests.ahk` (default true, FromMap defaults true on missing key); 2 in `settings_repository_tests.ahk` (round-trip both directions, missing key defaults to true); 4 in `settings_dialog_tests.ahk` (snapshot includes the field, restore replaces it, `ShowOutcomeBannerChanged` fires on flip). **Mudança A do plano original (rename `Cancel` → `SaveAsDnf`) foi deliberadamente pulada** — a investigação mostrou que `Commands.CancelRunRequested` é definido mas nunca publicado em código de produção (sem hotkey, sem botão, sem auto-trigger), apenas testado. Renomear sem entry point user-facing seria over-engineering pelo padrão do GSG §0 ("não trocar simplicidade por aparência de arquitetura"); o rename deve esperar até a ação existir como surface real.

- **PB display mode toggle (`cfg.pbDisplayMode`).** New Settings → DISPLAY checkbox swaps every PB-related surface in the overlay (the displayed value AND the live-timer color comparison target) between the all-time PB and the average of the latest five completed runs. Five widgets touched — Steve Plus, Compact Plus, Compact Classic, Steve Classic, Micro Plus — covering every layout that consults `_pbService` (only Micro Classic is excluded, since it doesn't read PB at all). Visual differentiation in `avg5` mode is per-widget: Steve Plus prefixes the bare value with a tilde (`~ MM:SS`, tilde = approximation), Compact Plus's block sub-labels swap `PB MM:SS` → `AVG MM:SS`, Compact Classic's line2 chip swaps `PB ZZ:ZZ / TT:TT` → `AVG ZZ:ZZ / TT:TT`, and the two color-only widgets (Steve Classic, Micro Plus) follow the new target silently — "current below target" still reads green either way, so the comparison semantic stays consistent across modes. Unlike `layoutVariant` (which requires a restart because widget GUIs are built once at boot), this flag is **hot-reloadable**: `SettingsDialog._OnSave` publishes the new `Evt.PbDisplayModeChanged` event, every PB-using widget subscribes, resets its derived caches (timer color, PB chip text) and calls `_Refresh` — no restart, no widget rebuild. New service `RunAverageService` (`src_v2/app/services/run_average_service.ahk`) does the math: pull-based mirror of `PersonalBestService` over `RunHistoryRepository`, with `GetAverageRunMs` / `GetAverageRunMsForAct(N)` / `GetAverageZoneMs(zone)` mirroring the three PB query surfaces. The window is hard-wired to `N_RECENT := 5` (single edit point for future UI surfacing). Per-act denominator counts only runs that *reached* that act (an Act-1-only run contributes to Act 1's average, not Act 3's); per-zone denominator counts only runs that *visited* the zone, and multiple visits within one run are summed before averaging — same shape as `ZoneTrackingService.GetZoneTotalWithActive`, which is what the PB comparison basis uses. Categories `loading` and `morte` are excluded from zone averages, matching the filter `PersonalBestService.RebuildFromHistory` applies. Two caches with separate lifetimes: run + per-act averages are built from `LoadSummaries(5)` (fast — meta + totals + checkpoints, no details), per-zone averages are built lazily on first `GetAverageZoneMs` after invalidation via `ListRunIds(5)` + `Load(runId)` per run (slow path — 5 INI reads — but bound to N_RECENT and cached until the next dirty flip). Cache invalidation subscribes to `Evt.RunCompleted` AND `Evt.RunCancelled` (both can write a new INI to disk; `RunCancelled` saves when the run lasted at least 3 minutes, so the safe superset re-loads on both) plus a manual `Invalidate()` hook for callers that change runs outside the finalize flow (run-history delete, import). Persistence: new `[Display]` INI section with key `PbMode=pb|avg5`; any value other than the literal `"avg5"` normalizes to `"pb"` in both `AppSettings.FromMap` and `SettingsRepository._LoadDisplay` (same defense-in-depth pattern `layoutVariant` uses). Defensive `_IsAvg5Mode()` helper in each widget performs a *dual check* — cfg present AND avgService injected AND `cfg.pbDisplayMode = "avg5"` — so a widget constructed without the service falls back to PB rather than render stale/empty values. Composition root (`src_v2/app/app.ahk`) constructs `RunAverageService` after `PersonalBestService`, threads it into all five widget construction sites as the final positional arg, and adds `"runAverage"` to `_AssertWired` so a regression that drops the wiring fails at boot. ToS compliance (GSG §18): the new service reads ONLY `data/runs/*.ini` (files this tracker itself wrote), with zero game interaction — no `Client.txt` read, no input simulation, no GGG API or website access — so the feature is **no risk** under PoE2 Terms of Use clauses 7(b–f, i). ~50 new tests across four files: 31 in the new `run_average_service_tests.ahk` (constructor type-checks, empty history, 1/2/5/7-run averages, per-act-by-act-reached, zone visit summation + cross-run averaging, defensive copies, cache invalidation via both events + manual hook, Dispose idempotency); 4 in `app_settings_tests.ahk` (default `"pb"`, accepted `"avg5"`, fallback on invalid, case-variation tolerance); 2 in `settings_repository_tests.ahk` (Display section round-trip, fallback on invalid INI); and 1 in `settings_dialog_tests.ahk` (publishes `PbDisplayModeChanged` with old/new modes on save). Spec lives in `PLUS_LAYOUTS_SPEC.md` §13; the placement reflects the Plus layouts being the largest PB-display surface, but §13.2 documents that the feature applies equally to Classic widgets.

- **Death Stats by Zone aggregate view.** New feature that answers "where do I die most often, across all play sessions" — surfaced via a "Death Stats" button in `RunStatsPlotDialog` (the run-statistics window, the natural home for cross-run aggregations). Three layers wire together at the composition root. (1) `DeathLogRepository` (`src_v2/infra/death_log_repository.ahk`) maintains an append-only `data/deaths.csv` with four columns (`ts`, `zoneName`, `patch`, `profile`). The file is created lazily on the first `Append`, so a fresh install does not carry an empty CSV. `Trim(zoneStr) = ""` rejects empty and whitespace-only zone names silently (a legitimate gap when a death fires before any `ZoneChanged` seeded the active zone); CR/LF in any field is rejected via the injected `WarningSink` (tag `[DeathLog]`) to prevent the same INI/CSV-corruption class of bug already documented for `RunHistoryRepository`. (2) `DeathStatsService` (`src_v2/app/services/death_stats_service.ahk`) is a pure read service: `Aggregate(filter := "")` returns `{totalDeaths, perZone, availablePatches, availableProfiles}` in one pass over the in-memory CSV. No cache — every call re-reads the file, which is fine because the dataset is small (one row per death across all play sessions) and the dialog is opened on demand. Town zones are dropped via `ZonesCatalog.IsTownName` when a catalog is wired (defensive against unknown zones — a zone the catalog doesn't recognize passes through to the chart so a new game patch doesn't silently hide the player's deaths). `availablePatches` / `availableProfiles` are extracted from the **whole** dataset, not the filtered subset, so the dialog's two dropdowns stay stable as the user cycles through filters; `perZone` is sorted by count desc via a stable insertion sort, and the available lists are sorted case-insensitively via `StrCompare(..., 1)`. (3) `DeathStatsDialog` (`src_v2/ui/death_stats_dialog.ahk`) subscribes to `Cmd.OpenDeathStatsRequested` and renders a ListView with three columns (Zone / Count / Bar). The Bar column is an ASCII proportion (`█` U+2588, max 30 chars) because AHK v2's ListView has no per-cell custom-draw hook short of subclassing via `WM_NOTIFY` (fragile and out of scope) — ASCII stays readable in the default font, copy-pastes cleanly into a Discord report, and survives HiDPI scaling without code. The two filter dropdowns use `"(All)"` as the no-filter sentinel (it's never a valid stored value: the upstream handler reads `cfg.gamePatch` / `cfg.profileName` which default to `"Unknown"` / `"Default"` on a fresh install, safe by inspection). Selection is preserved across `_Refresh` rebuilds and falls back to `"(All)"` if the previously-selected value vanished from the dataset.

  Composition-root wiring (`src_v2/app/app.ahk`) uses a **separate** subscriber for `Evt.DeathDetected` from the existing death-penalty handler (`LiveReconfigurationHandlers.ApplyDeathPenaltyToTimer`). The two have different concerns — live-timer adjustment versus append-only history persistence — and splitting them keeps each handler trivial to read and lets a future change to one path not risk the other. The new `_OnDeathDetectedForLog` reads the active zone via `ZoneTrackingService.GetActiveZone()` (canonical resolved name from the catalog, NOT a raw engine id — the §17 anti-regression item is preserved by construction), captures `cfg.gamePatch` and `cfg.profileName`, and calls `Append`. The event `data` parameter is ignored: `LogMonitorService`'s upstream filter already validated `character` against `cfg.characterName`, so by the time this fires we already implicitly own the run that died. The death log is **independent of run lifecycle**: deleting a run via `RunHistoryDialog` removes the `.ini` file and rebuilds PBs but does NOT retract the deaths that run contributed to `deaths.csv`, because the per-zone aggregate is meant to answer "where does the player die across all play sessions, ever" — coupling it to history would make the chart jump around every time the user prunes test aborts. Documented as a design constraint in `KNOWN_ISSUES.md` ("Death log is decoupled from run history").

  Adjacent changes carried in the same feature: the "Profile name" label in `SettingsDialog` is renamed to "Build" (vocabulary shift toward the PoE-player audience — a "build" is what they recognize); the underlying field is still `cfg.profileName`, and `AppSettings` / `SettingsRepository` / `DeathLogRepository` are unchanged — only the visible label moves. `build-dist.ps1` adds `data/deaths.csv` to `$ExcludeFiles` so a release zip never ships personal play data, and `tests_v2/build/test_build_dist.ps1` gains a sentinel row in `$mustFilter` proving the exclusion holds. 61 new tests across four files take the suite from 1874 to 1935: 18 in `death_log_repository_tests.ahk` (header round-trip, whitespace/empty rejection, CR/LF warn paths, malformed-row skip on Load, missing-file empty Load), 22 in `death_stats_service_tests.ahk` (aggregation correctness, filter combinations, town drop, stable sort on count ties, case-insensitive available sort, full-dataset available extraction), 4 in `speedkalandra_app_integration_tests.ahk` (constructor wiring, end-to-end DeathDetected → CSV row with active zone + patch + build, silent skip when no active zone, end-to-end aggregation via the production service), and 17 in `death_stats_dialog_tests.ahk` (constructor type-checks, `Cmd.OpenDeathStatsRequested` subscription, lifecycle idempotency, headless gate keeps `_gui = ""`, and the `FormatBar` static helper across zero/equal-max/half/clamp-above/min-one-block/zero-max-defensive/non-number-defensive cases).

- **Death Stats "All-time (from log)" view.** Same `DeathStatsDialog` gains a second data source: a button at the bottom-left toggles between the live CSV view (above) and a one-shot scan of the raw `Client.txt`. The all-time view is generated synchronously on each click, displayed in the same ListView (so the user reads the same Zone / Count / Bar layout regardless of source), and never persists — no cache survives the toggle to live, and a fresh `Open()` always starts in live. The toggle exists because the CSV-backed view only knows about deaths recorded since the user first installed SpeedKalandra, while many players' `Client.txt` reaches back months or years; the all-time scan surfaces that older history without ever writing it into the app's data directory. Three pieces wire together: (1) `DeathLogScanner` (`src_v2/app/services/death_log_scanner.ahk`, ~220 lines, 26 tests) is a pure standalone reader: receives a `logPath` and an optional `characterFilter`, streams the file via `FileOpen + while !AtEOF + ReadLine` (UTF-8 forced so non-ASCII zone names survive non-English Windows installs), parses two line types with regexes duplicated by design from `LogMonitorService._ExtractScene/Death` (header explains why duplication beats abstraction for two patterns), resolves zone names through `ZonesCatalog` the same way the live tail does, and returns `{success, errorMessage, totalDeaths, perZone, linesScanned, skippedNoZone}`. The `skippedNoZone` counter surfaces in the dialog total label as `(skipped: N outside any known zone)` when non-zero — typical when the log begins mid-session before any `[SCENE]` has been parsed. `_AssertWired` in the composition root now requires the scanner, catching wiring regressions at boot. (2) `DeathStatsDialog` gains `_mode` (`"live"` / `"alltime"`), `_alltimeResult`, `_scanner`, `_cfg` fields and the public hooks `GetMode()` / `IsInAlltimeMode()` / `ToggleAlltimeMode()` (the last exposed so headless tests can drive the toggle without firing the button OnEvent). The toggle reads `cfg.logFile` at call time, not at construction — the user may have changed the path in Settings since opening the dialog, and reading live avoids stale state. The view deliberately passes an **empty character filter** to the scanner instead of `cfg.characterName`: "all-time" is meant to include every character the player ever ran on the install, and current PoE2 builds do not emit `has been slain` for bosses (verified empirically against a real Client.txt that surfaced only 3 deaths when filtered to the active character versus the player's actual history of multiple alts), so dropping the filter does not pollute the result with boss kills. Failed scans (missing file, read error) leave the mode at live and surface a `MsgBox` (in production) or fail silently (in headless); successful scans flip the mode and call `_Refresh`, which disables the Patch/Build dropdowns (they're meaningless on the raw log) and rewrites the header to `"Death Stats - All-time (from Client.txt)"`. (3) An `Export...` button appears in alltime mode only — the live view's data is already at `data/deaths.csv` on disk where the user can copy it with a file manager, so a second export path would be redundant. The export delegates to a static `FormatExportCsv(perZone)` (semicolon-quoted UTF-8 via `CsvFile.FormatRow`, so the output is round-trippable through the same CsvFile parser the dialog otherwise reads) plus an `AtomicWriter.WriteAll` to the user-chosen path. The default folder is `%USERPROFILE%\Downloads` (falls back to `A_MyDocuments` if Downloads is absent — some corporate profiles remove it); the default filename embeds a timestamp so consecutive exports don't overwrite each other. 35 new tests take the suite from 1935 to 1970: 26 in `death_log_scanner_tests.ahk` (constructor type-check, path/file validation, basic parsing, character filter behaviour for both empty and non-empty, catalog effects including town drop and unknown-zone passthrough, no-zone edge case, sort + stability, the two static parsers); 9 in `death_stats_dialog_tests.ahk` (constructor type-checks for scanner + cfg, the mode-toggle lifecycle, an end-to-end assertion that the dialog ignores `cfg.characterName` and counts every death in the log, and the six `FormatExportCsv` cases including empty array, non-array defensive, single/multiple rows, quote escaping, internal-quote doubling); 1 added to the existing `speedkalandra_app_integration_tests.ahk::constructor_creates_death_log_components` to assert `deathLogScanner is DeathLogScanner` (the type check at the composition root mirrors the existing assertions for `deathLog` and `deathStatsService`).

- **Death Stats all-time view: cruel difficulty detection + campaign-only filter.** Empirical investigation of a real 115 MB Client.txt revealed two issues with the initial all-time view design: (1) the view was inflating counts with deaths in hideouts / atlas maps / endgame trials — noise that buries the campaign signal a speedrunner cares about, and (2) cruel-difficulty zones (the Acts 1–3 replay before endgame) were being **silently lost** because PoE2 does NOT emit `[SCENE] Set Source` for cruel zones — only `Generating level X area "<code>"`, with a `C_` prefix on the code to mark cruel. The view's only zone-detection path was the SCENE line, so cruel deaths were either attributed to the last normal zone seen before crossing into cruel (silent mis-attribution) or dropped as "before any zone" (silent loss). Both behaviours were verified against the real log — 25 `Generating level area "C_*"` events appeared with zero corresponding SCENE Set Source lines in the following 100 lines of each.

  Fix: `DeathLogScanner` gains a **third parser**, `_ParseAreaGen`, that extracts the area code from `Generating level X area "<code>" with seed N` lines and a new resolution path `_ResolveAreaCode(code)` that handles the `C_` prefix by stripping it, looking up the underlying internal id in the catalog, and appending `" (Cruel)"` to the display name when cruel was detected. The catalog itself does NOT gain cruel entries — the suffix is dynamic and works for every campaign zone (~77) without doubling the CSV. Scanner state machine now has three signals in precedence order: area-gen line (highest, the only cruel-aware signal), SCENE Set Source (fallback for normal-difficulty and when the area gen is truncated), and `has been slain.` (counted against the most recent resolved zone, or `skippedNonCampaign` if none). A SCENE that resolves to `""` (hideout, endgame, town, unknown) now resets `currentZone` so a death after leaving a campaign zone doesn't get mis-attributed to the campaign zone the player just left.

  Campaign-only filter: anything not in `data/zones.csv` is dropped at resolution time and the count surfaces in `skippedNonCampaign`, displayed in the dialog header as `(skipped: N outside campaign zones)`. This covers hideouts (e.g. `Canal Hideout`, code `HideoutCanal`), atlas maps (`Spider Woods`, `Sandspit`, codes `MapSeepage`/`MapAugury_NoBoss`), endgame trials (`The Burning Monolith`, code `P1_5`), and towns (`Clearfell Encampment`, including its cruel variant `C_G1_town`). The drop is transparent — the user sees both `Total deaths: N` and `(skipped: M outside campaign zones)`, so a low total never looks like a silent bug.

  Result-Map breaking change: the field `skippedNoZone` is renamed to `skippedNonCampaign` to reflect the broader semantics (it now counts every drop reason, not just "death before any [SCENE]"). `DeathStatsDialog._RefreshAlltime` and the scanner tests are updated accordingly. The static `_ParseScene` parser also gains an `Interlude` filter — PoE2 emits `[SCENE] Set Source [Interlude]` as a cinematic title card between cruel transitions, alongside `Act N` markers; both are now filtered as non-zone markers. Eight existing tests in `death_log_scanner_tests.ahk` were reworked (renamed for the new contract, assertions inverted where the campaign-only policy changed behaviour) and **eight new tests** were added: `scan_area_gen_resolves_normal_zone_via_internal_id`, `scan_area_gen_takes_precedence_over_following_scene`, `scan_detects_cruel_via_c_prefix_and_appends_suffix`, `scan_cruel_and_normal_same_zone_counted_independently`, `scan_drops_cruel_town_via_c_prefix_on_town_id`, `scan_drops_hideout_area_code_increments_skipped`, `scan_drops_endgame_map_area_code_increments_skipped`, `scan_hideout_after_campaign_resets_currentzone`, plus three new static-parser tests (`static_parse_area_gen_extracts_normal_code`, `_extracts_cruel_code_with_prefix`, `_returns_empty_for_non_matching`) and the `Interlude` filter on `static_parse_scene_returns_empty_for_null_unknown_act_marker_interlude`. The dialog tests are unchanged — all their fixtures use Mud Burrow / The Riverbank which are campaign zones, so the new policy is a no-op for them. Suite goes from 1970 to **1980** tests.

  Known limitation carried over: the **live tracking path** (`LogMonitorService`) still relies only on `[SCENE] Set Source` and therefore has the same cruel blind spot. A run that spans cruel difficulty will record deaths in `data/deaths.csv` attributed to the last normal zone before crossing the threshold. - **Operational caps on the import surface (gold-standard hardening).** `RunImportService` now refuses files above `MAX_FILE_BYTES = 10 MB` before any `FileRead`, using `FileGetSize` so a 100 MB hand-crafted JSON never consumes 100 MB+ of RAM just to fail later on schema parse. `RunExportFormat` gains six caps enforced inside `ValidateSchema`, with reasoning documented next to each constant: `MAX_RUNS_PER_FILE=5000` (a multi-league archive fits; a million-entry array does not), `MAX_STRING_LEN=500` (zone names cap at ~40 chars in the live catalog; profile/patch are user-defined; 500 leaves headroom for human input without admitting 10 MB of single-field payload), `MAX_DETAILS_PER_RUN=1000` (a 4-hour run at one zone change per 15 s = 960 details, plus per-loading rows), `MAX_TOTALS_PER_RUN=200` and `MAX_ZONE_PBS=200` (77 zones today; 200 absorbs future expansion), `MAX_ACT_CHECKPOINTS=20` (6 acts today; 20 absorbs future expansion). ### Changed

- **Overlay drag rewritten as event-driven (`WM_MOUSEMOVE` + `WM_LBUTTONUP`) with explicit `SetCapture`; Settings dialog frame slimmed.** The previous drag was a `SetTimer 16ms` polling loop calling `MouseGetPos` + `WinMove` on every tick. Under realistic load — five other `SetTimer` callbacks active on the AHK message pump (log monitor, focus auto-pause, tick emitter, loading detection, Ctrl poll) plus DWM compositing two AlwaysOnTop layered windows when the Settings dialog was open — the drag tick was preempted enough to produce visible stutter (effective ~30 fps with jitter instead of the 60 fps the period requested). The new path installs `OnMessage(WM_MOUSEMOVE)` inside `_OnLButtonDown` and removes it inside `_EndDrag`, so the dragged window moves only when the mouse actually moves (Windows already skips idle moves and a `_lastMouseX/Y` guard catches the rare coalesced replay). `OnMessage(WM_LBUTTONUP)` finishes the drag normally; a 100 ms `SetTimer` watchdog covers the lost-LBUTTONUP edge case (cross-process focus stolen mid-drag — capture is released by Windows when another process takes foreground, and the up event then goes nowhere). The watchdog also serves the `Stop()`/`UnregisterHwnd` cancellation paths via the shared idempotent `_EndDrag` + `_CleanupDrag` helpers, which release mouse capture and tear down the OnMessage handlers; cancellation paths skip `onDragEnd` while normal completion fires it, preserving the prior semantics for widget position persistence. `SetCapture(hwnd)` is called explicitly on LBUTTONDOWN because `_OnLButtonDown` returns 0 to suppress the click on child buttons (V1/V2/V3) — that suppression also blocks `DefWindowProc`, which is where Windows normally auto-captures the mouse on button-down; without explicit capture, `WM_MOUSEMOVE` stops arriving the moment the cursor leaves the overlay window, freezing the drag whenever the user moves faster than `WinMove` can keep up. `Start()` also force-sets `CoordMode("Mouse", "Screen")` so the drag handlers' `MouseGetPos` is in the same referential as the `WinGetPos`/`WinMove` calls (default `Client` would have silently produced wrong deltas when the active window changed mid-drag, and the same fix removes a latent bug in `_UpdateHoverState` which already compared screen-space `WinGetPos` against client-space `MouseGetPos`). The Settings dialog itself lost its `+Resize` flag in the same pass — the window has fixed dimensions (`WINDOW_W=560, WINDOW_H=620`) and never used sizing borders, but the resize-capable DWM frame is more expensive to repaint when an AlwaysOnTop layered window (overlay) crosses it during drag. The previous `static_drag_tick_ms_is_16` regression test is renamed to `static_drag_watchdog_ms_is_100` and the constant `DRAG_TICK_MS` is renamed to `DRAG_WATCHDOG_MS`; the interactive drag itself remains outside the headless test surface (real `OnMessage`/Win32 required), matching the `KNOWN_ISSUES.md` entry that already lists the drag/wheel paths as uncovered. The two changes (drag rewrite + Settings frame) are bundled because the Settings-open case was the trigger that exposed the polling cost; either alone would have helped less.

- **`GAME_EXECUTABLES` consolidated into `GameProcesses.POE2_EXECUTABLES`.** `FocusAutoPauseService` and `LoadingDetectionService` previously each declared the same hardcoded list of PoE2 executable names with a "keep in sync" warning in both files. The list now lives in `src_v2/domain/game_processes.ahk` and both services consume it directly. 6 new tests in `game_processes_tests.ahk` lock the contract (non-empty, every entry `.exe`, no duplicates, canonical standalone and Steam builds present).

- **`build-dist.ps1` release gate now genuinely waits for the AHK suite.** The previous form `& $ahkExe "tests_v2\run_tests.ahk"` was a silently broken gate when invoked from PowerShell 5.1 (which is what the `build-dist.bat` wrapper uses by default). AutoHotkey64.exe is Windows-subsystem, not console-subsystem, and PS 5.1 does not wait implicitly on GUI EXEs called via `&` — it is fire-and-forget. Empirical evidence: the build returned and produced the zip while the AHK process was still running the 50-second suite; the test summary printed *after* the zip was on disk. A null-defensive `$exitCode = $null ? 0 : raw` line, originally added because the CI runner sometimes returned `$null` from native processes, made the bug invisible by mapping the no-wait `$LASTEXITCODE = $null` to a treated-as-success 0. The gate is now `Start-Process -FilePath $ahkExe -Wait -PassThru -WindowStyle Hidden` plus `$testProc.ExitCode` (guaranteed populated by .NET's `Process.ExitCode` after `WaitForExit`, no null-defensive fallback needed); `-NoNewWindow` is deliberately omitted because it caused the original 19-minute hang on the GitHub Actions runner (inherited stdio handles leaving `WaitForExit` blocked on a pipe that never closes). A second layer of defense compares `tests_v2/tests_output.log`'s LastWriteTime against `$gateStartTime` captured before AHK launched — if AHK reports exit 0 but the log is stale, the build aborts loud ("the runner did not produce a fresh log during this build"). The CI workflow (`shell: pwsh`, PowerShell 7) keeps the `&` form because PS 7 *does* wait on GUI-subsystem EXEs invoked via `&`; the local-only nature of the bug is documented in the new comment block above the gate.

### Fixed

- **Hotkey-interrupted zone visit polluted PB totals (artificial sub-second zone PBs).** Pressing `Ctrl+Alt+F` to finalize a run shortly after entering a new zone left that zone with an artificially low PB candidate equal to the few seconds the player spent there before the hotkey. The same zone, in a normal run that crossed the transition out of it, would land 60–120 s — the interrupted-visit number replaced it because `RunSnapshotSaver` was passing the factual `zoneTotals` straight to `PersonalBestService.UpdateFromRun`, and the factual total at that point included the just-flushed elapsed of the interrupted visit. The fix tracks per-visit elapsed (`ZoneTrackingService._currentVisitMs`, accumulating in `_FlushActive` and resetting on every transition / lifecycle end), reads it from `RunSnapshotSaver.Save` via the new `GetCurrentVisitMs()` query in the `OnBeforeFinalize` hook (after `TimerStopped` flushes the visit but before `RunCompleted` publishes), subtracts it from a copy of the zone totals (`pbZoneTotals`), and passes that copy to `UpdateFromRun`. The factual `zoneTotals` is unchanged — the run's history file and post-run plot still show the full visit time as a historical fact; only the PB candidates exclude it. Permissive case preserved: if the zone had a prior complete visit in the same run (closed via transition) plus the interrupted one, only the interrupted visit's time is subtracted, the prior closed visit still counts as a PB candidate. Single-visit interrupted zones drop out of PB-eligible totals entirely (negative remainder → zone removed instead of recorded with a negative or zero PB). The `(interruptedZoneName, interruptedZoneVisitMs)` pair is persisted into the run's `[meta]` section by `RunHistoryRepository`, so `PersonalBestService.RebuildFromHistory` (the path that runs after `UndoLastSave` and after `Delete` from `RunHistoryDialog`) mirrors the same discount — Undo lands on the exact same PB as the original save, never on the inflated pre-fix one. Legacy runs persisted before this field existed load with the defaults `""` / `0` and skip the discount cleanly (no regression on previously-saved history). The visit accumulator is in-memory only by design (see `KNOWN_ISSUES.md` for the narrow crash + same-zone + hotkey edge case). 37 new tests: 14 on `ZoneTrackingService` (per-visit accumulation, pause/resume preservation, reset on every lifecycle path, finalize-sequence end-to-end), 8 on `RunSnapshotSaver` (discount applied, zone dropped when visit equals or exceeds total, keys injected into buildResult, cancelled runs don't call `UpdateFromRun`), 6 on `RunHistoryRepository` (INI round-trip, legacy default, defensive rejection of `\n` / `[` in `interruptedZoneName`), 7 on `PersonalBestService.RebuildFromHistory` (mirrored discount, drop-on-equal, defensive on exceed, no-op on empty/zero keys, only matching zone discounted, back-compat with legacy runs), and 2 integration tests in `speedkalandra_app_integration_tests.ahk` (original bug reproduction + permissive case) wiring the three layers through the real composition root.

- **Zone internal IDs leaking as human names (engine `G1_2` shown/saved instead of `Clearfell`).** PoE2's `Client.txt` emits `[SCENE] Set Source [<raw>]` with `<raw>` being either a human zone name ("Mud Burrow") or the engine's internal id ("G1_3"). The old `LogMonitorService` copied the raw text straight into `Events.ZoneChanged{zoneName: raw, sceneId: raw}`, so when the engine emitted the id, every downstream consumer keyed its data by the id: `ZoneTrackingService._totals["G1_3"]`, `RunStatsPlotBuilder._AddZoneDetails` finding `FindByName("G1_3")` empty (fell back to category="mapa", act=0, label="G1_3"), `RunHistoryRepository.Save` writing `[totals] G1_3=ms` to `data/runs/{runId}.ini`, and `PersonalBestService._zonePbs["G1_3"]` in `personal_bests.ini`. `GetActTotals()` silently dropped the zone from per-act tallies because `FindByName` of the id failed. The fix resolves the raw text to the canonical human name from `ZonesCatalog` at the single publisher (`LogMonitorService._ResolveZoneToHumanName`): try `FindByName` first (catches human-name lines, recovers canonical case), then `FindById` (catches engine-id lines, returns the catalog's display name), then pass through the raw text for unknown zones (a new game patch or randomized instance still surfaces in the overlay/history — just without act/isTown metadata). `Events.ZoneChanged.sceneId` continues to carry the raw text for diagnostics. The catalog is an optional constructor parameter on `LogMonitorService`; tests/headless setups without one preserve the pre-fix behaviour. The legacy `"You have entered <zone>"` parse branch is also routed through the resolver — not observed in current PoE2 `Client.txt` (every real-world `ZoneChanged` fires from `[SCENE]` today), retained defensively in case the engine reintroduces the line. 8 new tests across `log_monitor_service_tests.ahk` (resolution by id, resolution by name with case recovery, fallback for unknown zones, no-catalog passthrough, constructor type-check) plus 1 end-to-end guardrail in `speedkalandra_app_integration_tests.ahk` that proves `[SCENE] Set Source [G1_3]` lands in `zoneTracker._totals["Mud Burrow"]` (not `["G1_3"]`). Historical run files on disk that were saved with raw-id keys keep working (Load is unchanged); the user can `Reset PBs` or import-rebuild to regenerate clean PB keys from new runs.

- **INI-breaking characters in textual fields now rejected at import and on save.** Run files (`data/runs/{runId}.ini`) and the personal-bests INI rely on `\r`, `\n`, `[` and `]` having reserved structural meaning. Before this fix nothing stopped a hand-edited or maliciously crafted import payload from putting any of those characters into `profile`, `patch`, `firstTs`, `runId`, a zone-name key under `totals`, a detail row's `category`/`label`/`note`/`timestamp`, the PB `runPbRunId`, or a `zonePbs` zone key. A single `\n` would silently split a row across two INI lines on the next save, making the `[details].count` value mismatch the actual rows and corrupting `Load`. The fix is two layers: (1) `RunExportFormat.ValidateSchema` now scans every textual field on every run plus the PB block, and reports `"runs[N].fieldName contains INI-breaking character (\\n)"` so the user sees which field needs cleaning before the import touches disk; (2) `RunHistoryRepository._SerializeBuildResultToIni` defensively throws `ValueError` on the same character set, routed through the existing `WarningSink` so `Save` returns `false` and emits a warning identifying the failed `runId` — protection against a future parser regression smuggling bad text in through some other code path. Safe punctuation (`=`, `;`, `|`, parens, long strings, Unicode) is explicitly preserved by a round-trip test. 17 new tests: 9 covering each rejected field/character combination at the import layer (`run_export_format_tests.ahk`), 5 round-trip tests confirming safe characters survive Save/Load unchanged, and 3 defensive-throw tests confirming a bad field surfaces through the `WarningSink` instead of corrupting the file (`run_history_repository_tests.ahk`).

- **Removed accidental Exit hotkey (`Ctrl+Alt+Q`).** The entry script registered `^!q::ExitApp()` as a global hotkey. On ABNT2 and other international keyboards `AltGr+Q` is reported by Windows as `LCtrl+RAlt+Q`, which AHK's `^!` matches exactly — pressing AltGr+Q during a run closed the app and lost in-progress state. The tray menu "Exit" item already covers the use case, so the global hotkey was redundant and unsafe. Removed entirely; tray-menu Exit is the documented way to close the app.

- **`RunHistoryRepository.Save` is now atomic.** The previous implementation made roughly N direct `IniWrite` calls against `data/runs/{runId}.ini` (meta + totals + checkpoints + details, all sequential, each one a separate file open/modify/close round trip). A crash anywhere in that sequence — PC freeze, BSOD, mid-finalize close, antivirus quarantine grabbing the file — left a half-written run on disk: `[meta]` complete, `[totals]` missing, `[details]` count out of sync with the actual rows. The new implementation builds the full INI in memory and writes it through `AtomicWriter` (`.tmp` + `FileMove`), so the destination only ever sees a fully-formed file or no change at all. Same byte-compatible UTF-16 LE BOM output that `IniWrite` used to produce, so existing runs on disk and the `Load` path stay unchanged. The pure `_SerializeBuildResultToIni(buildResult, currentRunId, totalMs)` helper that builds the content is testable in isolation, and Save now also routes its failure path through the injected `WarningSink` instead of returning `true` regardless of what happened on disk. Two new tests cover the atomicity contract: no `.tmp` orphan after a successful save, and `Save` returns `false` + warns on writer failure.

- **Hydration ordering bug (RunStatsRecorder silently dropped hydrated runs).** `SpeedKalandraApp.__New` used to call `RunService.Hydrate` mid-construction. When the loaded state had an active run, `Hydrate` published `Evt.RunStarted{hydrated:true}` before `RunStatsRecorder` (and other downstream subscribers) had been created. Finalizing the hydrated run later produced a snapshot with `runId=""`, which `RunHistoryRepository.Save` silently rejected — the run was lost. `Hydrate` is now deferred to the end of `__New` after `_WireEventHandlers()`, and `ZoneTrackingService._OnRunStarted` respects the `hydrated:true` flag so the in-flight event no longer wipes the totals that were just restored from disk. Regression tests added in `zone_tracking_service_tests.ahk` and `speedkalandra_app_integration_tests.ahk`.
- **`LoadingTotalsService` wiped hydrated `_totalMs` on `RunStarted{hydrated:true}`.** Sibling of the hydration-ordering bug above: the composition root hydrates the loading total from disk via `loadingTotals.Hydrate(loadingMs)`, but `_handlerRunStarted` was hard-wired to `Reset()` and so the deferred `RunService.Hydrate()` event zeroed the value immediately after restoration. Loading time of an in-progress run was lost on every reload. Handler now goes through `_OnRunStarted(data)` which honours the `hydrated:true` flag, mirroring `ZoneTrackingService`. Regression tests `run_started_with_hydrated_flag_preserves_total_ms` and `run_started_without_hydrated_flag_wipes_total_ms` added in `loading_totals_service_tests.ahk`.
- **`LoadingDetectionService` false positives from non-game windows.** The default window provider matched the substring `"Path of Exile 2"`, which also matched Chrome tabs on the PoE2 wiki and Discord channels with that text. The HUD scanner then sampled pixels from the wrong window. Replaced with an `ahk_exe` scan over the same canonical list `FocusAutoPauseService` uses (`PathOfExile2Steam.exe`, `PathOfExile2_x64.exe`, etc.), and lock every follow-up `WinGetMinMax` / `WinGetPos` to the resolved HWND via `ahk_id`.
- **`UndoLastSave` left personal bests pointing at the deleted run.** Inconsistent with `RunHistoryDialog.Delete`, which already rebuilt PBs. `UndoLastSave` now invokes a new `_RebuildPbsFromHistory` helper after a successful delete (mirrors the dialog), so deleting the most-recent run via the tray immediately purges any PBs the deleted run contributed to.

- **`build-dist.ps1` rejects descendant destinations.** Running `.\build-dist.ps1 -DestDir ".\dist"` used to enter a recursive copy. Now both ancestor and descendant relationships between source and destination are rejected with a clear error.
- **CI reintroduced.** New `.github/workflows/test.yml` runs the `tests_v2/` suite on `windows-latest` for every push and PR; installs AutoHotkey v2 via Chocolatey, runs in headless mode (`SPEEDKALANDRA_TEST_NO_GUI=1`), and uploads `tests_output.log` as an artifact on failure. The previous attempt had been disabled because `#Warn All, MsgBox` in `run_tests.ahk` made the runner exit with a non-zero code even on all-green runs — routed to `OutputDebug` instead.

### Fixed

- **PB persistence failures no longer disguised as success.** Five mutations in `PersonalBestService` (`UpdateFromRun`, `Reset`, `LoadFromExternal`, `SetAsRunPb`, `RebuildFromHistory`) used to end with `try this._PersistToRepo()` — a bare silent try that swallowed both repository-returned-false (disk full, locked file, encoding error) and actual exceptions. The PB landed in memory but never reached `data/personal_bests.ini`, and the user saw nothing in `speedkalandra.log`. They now route through a new `_TryPersistOrWarn(context)` helper that surfaces both branches as visible `[PB]` WARNs while preserving the original behaviour of NOT breaking the finalize flow on persist failure. `_PersistToRepo` returns the bool from `_repo.Save` so the helper can distinguish "repo refused" from "repo threw".
- **`RunStateRepository.LoadZoneTotals` graded failure handling.** Used to silently return an empty Map on any `FileRead` failure, masking lock / permission / encoding errors that lose run-in-progress zone time. Now follows the graded taxonomy documented in `ARCHITECTURE.md` § 14: file missing is silent (expected new state); `FileRead` throws produces a `[RunState]` WARN with the exception; individual malformed lines are skipped silently (resilient to manual edits); a non-empty file where every line failed to parse produces a corrupt-file WARN.

### Changed

- **`WarningSink` introduced in `core/` for infra/services without a LogService dependency.** Repositories were logging Save / Delete / EnsureDir failures via `OutputDebug` only, which is invisible to the user without DbgView. The new `WarningSink` is a duck-typed observability channel (three implementations: `NullWarningSink`, `LogServiceWarningSink`, `InMemoryWarningSink`) injected into `PersonalBestRepository`, `RunStateRepository`, `RunHistoryRepository`, and `PersonalBestService` at composition-root construction. Each receives a tagged sink (`"PB"`, `"RunState"`, `"RunHistory"`) so a grep of `[PB]` / `[RunState]` / `[RunHistory]` in `speedkalandra.log` isolates failures by source layer. All `OutputDebug("X.Y failed: ...")` calls in those files were replaced with `this._warn.Warn(...)`. Default parameter is `""` so existing tests that construct repos with the legacy positional signature continue to pass (degrade to `NullWarningSink`). 41 new tests cover the sink contract, the persist-failure paths in `PersonalBestService` (with stub repos that return false / throw), and the graded `LoadZoneTotals` taxonomy.

### Documentation

- **`SECURITY.md` rewritten with explicit scope.** Replaces the minimal stub with a project-appropriate threat model (local-first, no network, no telemetry), enumerated in-scope categories (unexpected network activity, private data leakage, file writes outside the project directory, release packages containing personal files, malicious behaviour in distributed artifacts) and out-of-scope categories (incorrect timing, game-log format changes, AV false positives without reproducible unsafe behaviour, anything in forks). Disclosure preferences and the private channel are kept.
- **Issue templates added.** `.github/ISSUE_TEMPLATE/bug_report.md` and `.github/ISSUE_TEMPLATE/feature_request.md` plus a `config.yml` that disables blank issues and points security and known-limitations cases at the right docs. Bug template requests SpeedKalandra version, AHK version, Windows version, PoE2 client language, run state at the time, `EventTracingEnabled` flag, reproduction steps, expected vs actual, log snippet (with a privacy-review note), screenshot, and reproducibility category.

- **Error handling policy documented.** New § 14 in `ARCHITECTURE.md` formalizes the rule the codebase already follows for when `try` / `catch` should log on failure: *log where silence would hide data loss, broken state, or actionable failure*. Lists the four buckets every catch in the project falls into (logged via `LogService.Warn`; silent during lifecycle teardown; silent for cosmetic side effects; silent UI fallbacks that abort safely), with concrete examples and a decision tree for new code. Cross-referenced from § 13 (Conventions) and from `CONTRIBUTING.md`. One borderline silent catch found during the audit — `RunSnapshotSaver` reading the act-checkpoints map for the saved-run payload — was promoted to bucket (1) with an explicit `Warn` because a failure there would put an empty checkpoints map into the on-disk run file with no signal (data loss in disguise).

## [v0.1.1]

### Fixed

- **`TextEncoding.ConvertUtf16ToUtf8` and `MigrateIniToUtf8` removed** (Bug #2). The migration produced INIs with a UTF-8 BOM, and AHK v2's `IniRead(path, section, key, default)` silently returns the default on UTF-8 BOM files. Result: every key-based read after migration returned the fallback value — settings, run state, personal bests all looked freshly-installed. Reverted to UTF-16 LE BOM throughout. Documented in `text_encoding.ahk` and `ARCHITECTURE.md` §14.
- **"Lechtansi" bug: zone timer kept ticking during pause.** A `[SCENE]` line emitted by the game while alt-tabbed became a `ZoneChanged` event, which restarted `_startMs` in `ZoneTrackingService` even though `TimerPaused` had been observed earlier. Added a `_timerPaused` flag so `_OnZoneChanged` respects pause state.
