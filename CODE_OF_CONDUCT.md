# SpeedKalandra Code of Conduct

## Our commitment

SpeedKalandra is a personal, AI-assisted open source project built around practical Path of Exile 2 speedrun usage.

The goal of this project is to provide a useful, transparent, and maintainable tool for players who want better timing, run history, loading-zone tracking, personal best tracking, and local overlay support.

To keep the project welcoming and productive, everyone who participates in the SpeedKalandra community is expected to treat others with respect, patience, and good faith.

This Code of Conduct applies to all project spaces, including GitHub issues, pull requests, discussions, comments, documentation contributions, and any other communication channels officially associated with the project.

## Expected behavior

Participants are expected to:

- Be respectful and constructive.
- Assume good intent when possible.
- Give technical feedback without personal attacks.
- Keep discussions focused on the project, the code, the documentation, or the user experience.
- Be patient with people who may have different levels of technical knowledge.
- Welcome contributions, questions, bug reports, and suggestions from players and developers alike.
- Respect that this is a personal project maintained with limited time.
- Report bugs clearly, with enough detail to reproduce the issue when possible.
- Disagree constructively and explain technical concerns with reasoning.

## Unacceptable behavior

The following behavior is not acceptable:

- Harassment, intimidation, or threats.
- Personal insults, name-calling, or hostile comments.
- Discriminatory language or behavior.
- Sexualized language, imagery, or unwanted attention.
- Posting or threatening to post someone else’s private information.
- Repeatedly derailing technical discussions.
- Trolling, bad-faith arguments, or intentionally wasting maintainer time.
- Aggressive demands for support, features, or deadlines.
- Encouraging any of the above behavior.

## Technical criticism

Technical criticism is welcome.

This project is intentionally transparent about being AI-assisted. That does not mean the project is above criticism. Bugs, design issues, architecture concerns, maintainability problems, and documentation mistakes should be reported honestly.

However, criticism should focus on the work, not the person.

Examples of acceptable feedback:

- “This function has too many responsibilities.”
- “This error path can fail silently.”
- “This test does not cover the regression it claims to cover.”
- “This documentation is outdated.”
- “This design may be hard to maintain.”


Strong technical opinions are fine. Personal attacks are not.

## Reporting issues

If you believe someone is violating this Code of Conduct, please report it by contacting the project maintainer.
